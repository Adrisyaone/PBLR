161  Level 13 Capstone: A Resilient File-Reading Toolkit

The Problem

Level 10’s automation capstone combined all 5 clinic files with list.files() and a for loop — but it assumed every file would be present and well-formed. Your professor wants a genuinely production-ready version: one that uses a regex pattern to find files (Page 7), skips unreadable ones gracefully instead of crashing (Page 4), and returns a clean, custom-printed summary object (Page 3) reporting exactly what happened — not just the combined data.

Your Mission

Combine this level’s tools — closures, S3 objects, error handling, and regex — into one resilient, reusable function that reads a folder of files and reports on its own success.

Think Before Coding

  • Level 10’s capstone used list.files(pattern = "\\.csv$") to find files. If one of those files were deleted or corrupted, would the original loop keep going, or crash entirely?
  • If a function’s return value needs to report both the combined data and metadata about what succeeded or failed, would a single data frame be enough, or would a custom S3 object (Page 3) fit better?

161.1 The R Concept

This capstone’s function combines four separate ideas from across this level into one:

read_clinic_folder <- function(folder_path) {
  csv_files <- list.files(folder_path, pattern = "\\.csv$", full.names = TRUE)  # Page 7 (regex)

  results <- vector("list", length(csv_files))
  failed <- character(0)

  for (i in seq_along(csv_files)) {
    results[[i]] <- tryCatch({                    # Page 4 (error handling)
      read.csv(csv_files[i])
    }, error = function(e) {
      failed <<- c(failed, csv_files[i])
      NULL
    })
  }

  successful <- results[!sapply(results, is.null)]

  structure(list(                                  # Page 3 (S3 object)
    data = do.call(rbind, successful),
    n_files_found = length(csv_files),
    n_files_read = length(successful),
    failed_files = failed
  ), class = "clinic_batch")
}

print.clinic_batch <- function(x, ...) {
  cat(sprintf("Clinic batch: %d/%d files read successfully (%d rows total)\n",
              x$n_files_read, x$n_files_found, nrow(x$data)))
  if (length(x$failed_files) > 0) {
    cat("Failed files:", paste(x$failed_files, collapse = ", "), "\n")
  }
}

161.5 Interpret the Result

This capstone function is a genuine step up from Level 10’s automation: it doesn’t just automate a repeated task, it does so resiliently, reporting exactly what succeeded and what didn’t through a clean, custom-printed S3 object rather than a raw data frame or a script that halts on the first problem. This combination — regex-based file discovery, graceful error handling, and an informative custom result type — is close to what a genuinely reusable, shareable R function (Page 10’s package structure) should look like.

Common Mistakes

  • Using <- instead of <<- inside the tryCatch() error handler. failed <<- c(failed, csv_files[i]) uses the super-assignment operator (Page 1’s environments) specifically because the error handler runs in its own function environment — a plain <- would create a new local failed variable instead of updating the one the loop can see afterward.
  • Forgetting do.call(rbind, successful) fails if successful is empty. If every single file fails to read, successful is an empty list, and do.call(rbind, list()) produces NULL rather than an error — code using batch$data afterward should check for this case before assuming it’s a proper data frame.
  • Not testing the failure path at all. It’s tempting to only test a function like this against files that work — deliberately testing against a missing or malformed file (as the Explore section suggests) is the only way to confirm the error-handling logic actually behaves as intended.

161.8 Key Takeaways

  • A genuinely resilient automation function combines several ideas at once: pattern-based file discovery (regex), graceful error handling (tryCatch()), and an informative, custom-printed result object (S3) — rather than any single technique alone.
  • <<- (super-assignment) is often necessary inside a tryCatch() error handler or other nested function, specifically to update a variable in an enclosing environment rather than creating a new local one.
  • Testing a function’s failure path deliberately (not just its success path) is the only reliable way to confirm error-handling logic actually works as intended.
