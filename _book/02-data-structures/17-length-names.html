37  length() and names()

The Problem

A new intern joins your team and, trying to be helpful, checks the size of the participants dataset by running length(participants). They report back: “It only has 8 things in it!” — sounding alarmed, since they expected 100. You need to explain what actually just happened, and show them the right tool for what they were really trying to check.

Your Mission

Understand what length() actually measures on different object types, and use names() to inspect or set labels on vectors, lists, and data frames alike.

Think Before Coding

  • You learned two pages ago that a data frame is really a collection of columns, each one a vector. If length() measures “how many things are in this container,” what would “things” mean for a data frame specifically — rows, or columns?
  • If names() can retrieve labels from a named vector (earlier in this level), might it also retrieve something meaningful from a data frame or a list?

37.1 The R Concept

length() measures the number of top-level elements in an object — but what counts as a “top-level element” depends entirely on the structure:

length(vector)      # number of individual values
length(list_obj)     # number of top-level list elements (as you saw last-but-one page)
length(data_frame)   # number of COLUMNS - because a data frame is a list of columns!

This last one is the trap the intern fell into: a data frame is, structurally, a special kind of list where each column is one element — so length(participants) returns 8 (the number of columns), not 100 (the number of rows). For row count, you need nrow() from the previous page, not length().

names() retrieves (or sets) the labels attached to an object’s top-level elements:

names(vector)       # element names, if any (as with named vectors)
names(list_obj)      # the list's element names
names(data_frame)    # the column names

37.2 Demonstration

participants <- read.csv("data/participants.csv")

length(participants)
[1] 8
nrow(participants)
[1] 100
names(participants)
[1] "id"           "age"          "sex"          "county"       "systolic_bp"
[6] "diastolic_bp" "bmi"          "smoker"
# length() on the age COLUMN (a vector) behaves as expected
length(participants$age)
[1] 100

37.3 Explore

Run length(participants$age) versus length(participants) side by side, and notice they answer two genuinely different questions: “how many values are in this one column?” versus “how many columns does this whole data frame have?” Then try names(participants$age) — since age is a plain, unnamed vector, what do you think this returns?

37.4 Solve the Problem

# What the intern actually needed:
nrow(participants)    # 100 - the number of rows (participants)
[1] 100
# What length() actually measured:
length(participants)  # 8 - the number of columns (variables)
[1] 8
names(participants)   # the 8 column names, confirming what's being counted
[1] "id"           "age"          "sex"          "county"       "systolic_bp"
[6] "diastolic_bp" "bmi"          "smoker"

You can now explain it clearly to the intern: “length() on a data frame counts columns, not rows, because a data frame is really a list where each column is one element. For row count, always reach for nrow() instead.”

37.5 Interpret the Result

The dataset genuinely does have 100 participants (rows) and 8 variables (columns) — both facts are true at once, and length() and nrow() simply answer different questions about the same object. Neither function is “wrong”; the intern’s surprise came from expecting length() to answer a question it was never built to answer for a two-dimensional structure.

Common Mistakes

  • Using length() when you mean nrow(). This is, by far, the single most common version of this mistake in real R code — always reach for nrow() for row counts on a data frame, and reserve length() for vectors, lists, or explicitly checking a data frame’s column count.
  • Forgetting a data frame is structurally a list. This single fact explains both why length(df) counts columns and why df$column and list-style [[ ]] access work identically on data frames — they share the same underlying structure.
  • Assuming names() always returns column names. On a plain, unnamed vector (like participants$age), names() returns NULL — there’s nothing to name individual elements of an ordinary numeric column.

37.6 Practice Problem

Confirm, using length() and nrow() together in the same check (with &, as in the dimensions page), that participants still has 8 columns and more than 90 rows.

37.7 Challenge Problem

Build a list patient <- list(id = 1, age = 45, vitals = c(120, 80)). Predict, before running any code, what length(patient) will return, and separately what length(patient$vitals) will return — explain why these two numbers are different even though both trace back to the same object.

37.8 Key Takeaways

  • length() counts top-level elements — for a plain vector, that’s individual values; for a data frame (structurally a list of columns), that’s the number of columns, not rows.
  • For row counts on a data frame, always use nrow(), not length().
  • names() retrieves the labels on a vector’s elements, a list’s elements, or a data frame’s columns — what it returns depends on what labels actually exist on that particular object.

37.9 Where you’ve arrived

You started this level holding one value at a time. You can now build and index vectors every which way, convert messy categories into properly ordered factors, construct and slice data frames by row and column, work with matrices and lists, understand tibbles as data frames with extra guardrails, and — on this very last page — correctly distinguish what length() actually measures on different kinds of objects, instead of falling into the same trap the intern did.
