38  Reading CSV Files with readr

The Problem

Base R’s read.csv(), which you learned in Level 1, works — but it’s slow on large files, and it silently makes some decisions you might not want (like converting text columns in ways that vary subtly across R versions). A colleague mentions they always use read_csv() instead, from a package called readr, and that it “just tells you upfront what it did.” You want to know what’s actually different.

Your Mission

Import a CSV file using readr::read_csv(), and understand how it differs from base R’s read.csv().

Think Before Coding

  • From Level 1, what did read.csv() return, and what functions did you use afterward to check it imported correctly?
  • If a function “tells you upfront what it did” while importing, what kind of information would be most useful to see immediately?

38.1 The R Concept

read_csv(), from the readr package (part of the tidyverse), reads CSV files with a few meaningful differences from base R’s read.csv():

library(readr)
my_data <- read_csv("path/to/file.csv")
  • It returns a tibble (Level 2), not a plain data frame — with all the safety benefits you already know about.
  • It prints a column specification message immediately after import, showing exactly what type it guessed for each column — surfacing potential problems immediately, rather than requiring you to run str() separately to find out.
  • It’s meaningfully faster on large files, which matters once your datasets grow beyond a few hundred rows.
  • Unlike base R historically, it never converts character columns to factors automatically, and consistently treats blank fields as missing across all column types (a subtlety you’ll see matters a lot on the missing-values page later in this level).
participants <- read_csv("data/participants.csv")
Rows: 100 Columns: 8
── Column specification ────────────────────
Delimiter: ","
chr (3): sex, county, smoker
dbl (5): id, age, systolic_bp, diastolic_bp, bmi

ℹ Specify the column types or set `show_col_types = FALSE` to quiet this message.

38.4 Solve the Problem

library(readr)

participants <- read_csv("data/participants.csv")
Rows: 100 Columns: 8
── Column specification ────────────────────
Delimiter: ","
chr (3): sex, county, smoker
dbl (5): id, age, systolic_bp, diastolic_bp, bmi

The column specification message immediately confirms, without running any additional code, that age, systolic_bp, diastolic_bp, and bmi were all correctly detected as numeric (dbl), and sex, county, and smoker were correctly detected as text (chr) — exactly what you’d expect, visible upfront rather than requiring a separate str() call to discover.

38.5 Interpret the Result

Seeing the column specification message print immediately after import means read_csv() guessed every column’s type correctly on this particular file — a genuinely useful sanity check that base R’s read.csv() doesn’t surface automatically. Later in this level, on a messier file, you’ll see this same message actively catch a column that didn’t import the way you’d expect.

Common Mistakes

  • Forgetting to load readr (or tidyverse). Just like any package, read_csv() requires library(readr) (or library(tidyverse)) first — otherwise you’ll get the familiar could not find function error from Level 1.
  • Ignoring the column specification message. Treating it as noise to scroll past, rather than as a free, automatic check on whether each column imported as the type you expected.
  • Assuming read.csv() and read_csv() are interchangeable in every respect. They return different object classes (data frame vs. tibble) and differ in a few default behaviors — mixing them up mid-script can cause subtle inconsistencies.

38.8 Key Takeaways

  • readr::read_csv() is a modern, faster alternative to base R’s read.csv(), returning a tibble and printing a column specification immediately after import.
  • The column specification message is a free, automatic type-check — always worth reading, not scrolling past.
  • A tibble from read_csv() is still fundamentally a data frame, exactly as you learned in Level 2.
