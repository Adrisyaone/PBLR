84  Pivoting Multiple Measurements at Once

The Problem

Back on the pivot_longer() page, a Common Mistake warned that pivoting only sbp_visit1/sbp_visit2/sbp_visit3 leaves weight_visit1/weight_visit2/weight_visit3 untouched — duplicating each participant’s weight values across all three of their new long rows. Your professor actually wants both measurements reshaped together, correctly: one row per participant-visit, with separate sbp and weight columns side by side.

Your Mission

Reshape multiple measurement types to long format simultaneously, using pivot_longer()’s special .value token.

Think Before Coding

  • Look at the column name sbp_visit1. It actually encodes two pieces of information at once — which measurement, and which visit. Can you identify each piece?
  • If pivot_longer() could split a column name into pieces the same way separate() splits a column’s values, what would you want the “measurement” piece to become, versus the “visit” piece?

84.1 The R Concept

Column names like sbp_visit1 and weight_visit1 encode two pieces of information joined by an underscore: which measurement (sbp or weight), and which visit (visit1). The special .value token in names_to, combined with names_sep, tells pivot_longer() to split each column name into pieces and use one specific piece (wherever .value is placed) as the name of a whole new column, rather than folding it into a single generic value column:

pivot_longer(df,
  cols = matches("visit"),
  names_to = c(".value", "visit"),
  names_sep = "_"
)
pivot_longer(bp_wide,
  cols = c(starts_with("sbp_visit"), starts_with("weight_visit")),
  names_to = c(".value", "visit"),
  names_sep = "_"
)

Here, sbp_visit1 splits at the underscore into "sbp" and "visit1". Because .value occupies the first position in names_to, that first piece ("sbp") becomes an actual column name in the result, while the second piece ("visit1") becomes a value in a visit column — giving you sbp and weight as two separate, properly reshaped columns, instead of duplicating one across the other.

84.2 Demonstration

library(tidyr)

bp_wide <- read.csv("data/bp_visits_wide.csv")

bp_long_both <- pivot_longer(bp_wide,
  cols = c(starts_with("sbp_visit"), starts_with("weight_visit")),
  names_to = c(".value", "visit"),
  names_sep = "_"
)

names(bp_long_both)
[1] "id"          "county"      "clinic_site" "visit"       "sbp"
[6] "weight"

Both sbp and weight now exist as genuine, separate columns — not one duplicated across the other’s long rows.

head(bp_long_both[, c("id", "visit", "sbp", "weight")], 6)
  id  visit sbp weight
1  1 visit1 125  101.4
2  1 visit2 120  100.8
3  1 visit3 117  101.0
4  2 visit1 123   80.1
5  2 visit2 121   79.6
6  2 visit3 129   79.0

84.3 Explore

Try nrow(bp_long_both). It should still be 84 (28 participants × 3 visits) — reshaping two measurements together doesn’t create extra rows, since .value correctly keeps sbp and weight as separate columns within the same row, rather than stacking them into additional rows.

84.4 Solve the Problem

library(tidyr)

bp_wide <- read.csv("data/bp_visits_wide.csv")

bp_long_both <- pivot_longer(bp_wide,
  cols = c(starts_with("sbp_visit"), starts_with("weight_visit")),
  names_to = c(".value", "visit"),
  names_sep = "_"
)

nrow(bp_long_both)
[1] 84
sum(!is.na(bp_long_both$sbp))
[1] 82
sum(!is.na(bp_long_both$weight))
[1] 82

84 rows (not 168), with both sbp and weight correctly present as their own columns in every row — exactly the properly reshaped structure your professor asked for, with no duplication of one measurement across the other.

84.5 Interpret the Result

bp_long_both is now genuinely “tidy”: one row per participant-visit combination, with each measurement type in its own column. This is the format that would let you group_by(visit) and compute means of both sbp and weight in a single summarise(across(...)) call (Level 5), something the earlier single-measurement reshape couldn’t offer without either duplicating data or reshaping twice.

Common Mistakes

  • Forgetting to include .value in names_to. Without it, pivot_longer() has no way to know that "sbp" and "weight" should become separate columns rather than both being lumped into one undifferentiated value column.
  • Choosing the wrong names_sep. sbp_visit1 splits correctly on the first underscore into "sbp" and "visit1" — but if a measurement name itself contained an underscore, the split would produce more pieces than the two names_to expects.
  • Reshaping each measurement separately when they didn’t need to be. Doing two separate pivot_longer() calls (one for sbp, one for weight) and then trying to merge them back together is far more error-prone than using .value to reshape both in a single step.

84.6 Practice Problem

Confirm that bp_long_both’s sbp column, restricted to visit == "visit1", matches bp_wide$sbp_visit1 exactly (hint: use filter() from Level 5, then compare with identical() after matching row order by id).

84.7 Challenge Problem

Use bp_long_both, group_by(visit), and summarise(across(c(sbp, weight), ~ mean(.x, na.rm = TRUE))) (Level 5’s across()) to compute mean blood pressure and mean weight for each visit in a single call.

84.8 Key Takeaways

  • The .value token in pivot_longer()’s names_to lets you reshape multiple measurement types into separate columns at once, rather than lumping them all into a single generic value column.
  • names_sep (or names_pattern) tells pivot_longer() how to split each compound column name into its .value and other pieces.
  • Reshaping multiple measurements together in one call avoids both row-count duplication and the awkwardness of merging separately reshaped pieces back together.

    4. Has no special meaning; it’s just a placeholder string

  3. Predict the output. After the .value-based reshape, how many columns would bp_long_both have among id, county, clinic_site, visit, sbp, weight?

  4. Find the error.
